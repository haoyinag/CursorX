# 生成升级检查清单

目标：当当前版本引入安装方式、命令集、配置片段或工作流变化时，生成一份面向使用者的升级检查清单。

用户可以在命令后补充升级对象，例如：

- `/upgrade-checklist`
- `/upgrade-checklist for teams using global slash commands`

其中 `${user_input}` 是目标升级对象或需要强调的升级场景，可以为空。

## 必须遵守

1. 只基于真实变化生成升级清单，不要虚构升级步骤。
2. 升级检查清单必须优先回答：
   - 谁需要关注这次升级
   - 需要执行哪些动作
   - 如何验证升级完成
3. 如果本次改动对现有用户无升级动作要求，要明确写“无需额外升级操作”。
4. 如果 `${user_input}` 非空，应优先围绕该对象组织检查清单。
5. 检查清单要可执行，不要只写抽象建议。

## 执行步骤

1. 先阅读当前版本对应的差异、相关 README 和命令说明。
2. 识别是否涉及：
   - 新命令安装
   - 配套脚本同步
   - 旧命令替换
   - 文档或使用方式变化
3. 按用户动作拆成 checklist。
4. 补一段升级完成后的验证方式。

## 输出格式要求

- `Who should upgrade`
- `Checklist`
- `Verification`

其中 `Checklist` 必须使用可勾选的步骤。

## 适用边界

- 适合命令安装器升级、命令集扩充、目录或流程变化后的升级说明
- 不适合替代 release notes；它更偏“用户接下来要做什么”
