# P2 路线选项

这份文档用于把 `CursorX` 在 `P2` 阶段可能出现的几条路线摊开比较，避免后续只凭直觉切换方向。

## 评估前提

当前已知事实：

- `slash-commands/` 是主产品层
- `cursorx-cli` 已发布到 npm，并且已经多次真实迭代
- 内容层已经具备持续浏览价值
- 但还没有充分证据证明必须立刻切换产品形态

## 路线 A：继续保持单仓，但强化发布治理

### 路线 A 适合前提

- npm 包与内容仓仍然高度耦合
- 用户心智还主要围绕“CursorX 总仓库”
- 主要问题是发布节奏和版本管理，而不是仓库边界本身

### 路线 A 主要动作

- 继续维持单仓
- 强化 `CHANGELOG`、release process、版本同步规则
- 继续让 `cursorx-cli` 直接复用 `slash-commands/index.json`

### 路线 A 优点

- 维护成本最低
- 用户理解成本最低
- 不会引入新的跨仓同步问题

### 路线 A 风险

- 如果 CLI 发布开始明显更频繁，单仓会逐渐显得笨重
- 包级和仓库级变更仍可能相互牵制

## 路线 B：把 `cursorx-cli` 逐步独立成更明确的产品边界

### 路线 B 适合前提

- npm 包更新开始明显快于内容层
- 用户有一部分只关心安装和分发，不关心内容仓库
- 包级 changelog、release notes、问题管理需求越来越强

### 路线 B 主要动作

- 先不急着拆仓，但先把包级治理做完整
- 逐步强化 `cursorx-cli` 的独立版本叙事
- 在满足阈值后，再评估是否真的拆出第二仓库

### 路线 B 优点

- 能逐步验证“独立产品边界”是否真的存在
- 比直接拆仓更稳
- 仍保留和内容仓的强连接

### 路线 B 风险

- 可能进入“半独立”状态一段时间
- 需要更清晰地区分仓库 changelog 和包 changelog

## 路线 C：开始探索 Plugin / Marketplace 形态

### 路线 C 适合前提

- 用户开始更关注一键安装和编辑器内发现能力
- CLI 已经不是唯一最佳分发路径
- 你希望把 `CursorX` 从命令集合进一步做成产品入口

### 路线 C 主要动作

- 调研 Cursor Plugin / Marketplace 的能力边界
- 明确哪些内容适合插件化，哪些仍适合保留在仓库层
- 先做探索，不急着替换现有 CLI 和仓库结构

### 路线 C 优点

- 可能获得更自然的安装体验
- 更贴近“产品入口”心智

### 路线 C 风险

- 当前证据最少
- 容易过早转移注意力
- 可能让还没完全稳定的命令层再次被重构

## 当前建议

当前最合理的默认路径是：

1. 先采用路线 A
2. 同时保留路线 B 的观测准备
3. 仅把路线 C 视为探索项，而不是当前主线

## 为什么现在不建议直接跳到路线 B 或 C

- 当前虽然已经出现包级版本和 changelog，但还没有足够强的“独立产品压力”
- 当前的主问题更像治理收束，而不是结构失效
- 如果现在直接切形态，容易在“内容还在继续迭代”的情况下引入额外复杂度
