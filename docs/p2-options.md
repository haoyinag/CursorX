# P2 路线选项

把 P2 阶段可能的几条路摊开比较，避免凭直觉切方向。

## 前提（当前已知）

- `slash-commands/` 是主产品层  
- `cursorx-cli` 已在 npm 上多次真实迭代  
- 内容层已值得持续浏览  
- 尚无充分证据表明 **必须立刻** 切换产品形态  

## 路线 A：单仓 + 强化发布治理

**适合**：npm 与内容仓仍高度耦合；用户心智仍以「CursorX 总仓库」为主；痛点在发布节奏与版本管理，而非仓库边界。

**主要动作**：维持单仓；强化 `CHANGELOG`、release process、版本同步；`cursorx-cli` 继续直接复用 `slash-commands/index.json`。

**优点**：维护成本与用户理解成本最低；无新增跨仓同步。

**风险**：若 CLI 发布明显更频繁，单仓会变笨重；包级与仓库级变更仍可能互相牵制。

## 路线 B：`cursorx-cli` 逐步独立为更清晰的产品边界

**适合**：npm 更新明显快于内容层；部分用户只关心安装与分发；包级 changelog、release notes、issue 管理需求变强。

**主要动作**：先不拆仓，把包级治理做完整；逐步强化 `cursorx-cli` 的独立版本叙事；满足阈值后再评估是否真要第二仓库。

**优点**：可逐步验证「独立产品边界」是否真实存在；比直接拆仓稳；仍与内容仓强连接。

**风险**：可能长期处于「半独立」；需更清晰区分仓库 changelog 与包 changelog。

## 路线 C：探索 Plugin / Marketplace

**适合**：用户更关注一键安装与编辑器内发现；CLI 不再是唯一最佳分发路径；希望把 CursorX 从命令集合升级为产品入口。

**主要动作**：调研 Cursor Plugin / Marketplace 边界；明确哪些适合插件化、哪些保留在仓库；先探索，不急于替换现有 CLI 与仓库结构。

**优点**：可能获得更自然的安装体验；更接近「产品入口」心智。

**风险**：当前证据最少；容易过早吸走注意力；可能让尚未稳定的命令层再次被大改。

## 当前默认路径

1. 先采用路线 A  
2. 同时保留路线 B 的观测准备  
3. 路线 C 仅作探索项，**不是**当前主线  

若要把路线 B 从「备选」推进为「可执行」，继续读 [`p2-route-b-thresholds.md`](./p2-route-b-thresholds.md)。

## 为何不建议现在直接跳到 B 或 C

- 已有包级版本与 changelog，但「独立产品压力」仍不够强  
- 主问题更像治理收束，而非结构失效  
- 内容仍在迭代时硬切形态，容易叠加复杂度
