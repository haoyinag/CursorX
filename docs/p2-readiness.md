# P2 进入前检查

这份文档用于判断 `CursorX` 是否应该从当前的 `P1` 收口状态，进入 `P2` 的评估阶段。

## 先澄清什么是 P2

当前这里说的 `P2`，不是“继续补内容”，而是开始认真评估这些问题：

- 是否进入更明确的 release 节奏管理
- 是否拆出更独立的产品仓或 changelog 体系
- 是否进入 Cursor Plugin / Marketplace 方向

也就是说，`P2` 更偏产品治理和分发形态，而不是继续补第一批资产。

## 进入 P2 前的门槛

只有当下面问题大多回答为“是”时，才建议进入 `P2`：

1. 主产品层是否已经稳定
   - `slash-commands/` 是否已经覆盖一条完整核心工作流
   - 安装、自检、文档和发布语义是否已经稳定
2. 内容层是否已经成型
   - `commands/` 是否已经像参考层，而不是零散样板
   - `skills/`、`tips/`、`configs/` 是否已经具备持续浏览价值
3. 发布节奏是否开始独立
   - `cursorx-cli` 的发版是否开始频繁独立于仓库内容层
   - 是否已经出现单独维护包级 changelog 或 release notes 的需要
4. 用户心智是否开始分化
   - 是否有明显用户只关心安装器，而不关心内容仓库
   - 是否需要把“安装工具”和“内容总仓”分开沟通
5. 当前单仓是否开始形成负担
   - README、索引、发布流程是否开始频繁跨层同步
   - 是否已经出现单仓带来的理解或维护成本

## 当前判断

截至当前状态：

- `P1` 已接近完成，且核心工作流已经成型
- `cursorx-cli` 已多次真实发布，说明分发链路成立
- 但还没有足够证据证明现在就必须拆出独立产品形态

所以当前更合理的结论是：

- 可以开始“评估 P2”
- 但不建议立刻进入大规模形态切换

## 推荐动作

如果你准备进入 `P2`，建议按这个顺序：

1. 先观察一段时间：
   - npm 包更新频率
   - 仓库内容更新频率
   - 用户最关注的是安装还是内容
2. 再决定是否需要：
   - 单独维护 `cursorx-cli` 的 changelog / release page
   - 明确双仓策略
   - 评估 Plugin / Marketplace 方案
3. 在做这些动作前，尽量不要再大规模扩内容层目录

## 当前不建议做的事

- 不建议为了“进入下一阶段”而强行拆仓
- 不建议在证据不足时过早切到 Plugin / Marketplace 主叙事
- 不建议一边继续大规模扩内容，一边同时改产品形态
