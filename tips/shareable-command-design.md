# 先写可分享命令，再写项目私有规则

## 适用场景

- 你准备新增一个 slash command
- 你不确定它应该是通用命令还是项目命令
- 你担心把私有约束直接写死进公共命令

## 具体步骤

1. 先问自己：这个命令是否能跨项目复用。
2. 如果答案是可以，就优先把它写成通用命令。
3. 如果某些规则只属于单一团队或仓库，就把它们拆到项目模板或项目私有命令。
4. 在命令说明里明确：
   - 目标场景
   - 适用边界
   - 哪些规则是通用的
   - 哪些规则应该由项目自己注入

## 效果说明

- 能保持 `CursorX` 的利他属性
- 有助于把仓库做成真正可复用的命令产品层
- 能减少公共命令被单一项目绑架的问题

## 延伸建议

- 可先看 `slash-commands/project/git-push.project-template.md`
- 新增命令前，建议先对照 `docs/roadmap-status.md` 的三条自检
